## 拷问对象

来自 Stage 1 用户选定的 **A + C** 两个方向：

- **A. 极简一页静态名片**（约束：纯静态 / 无后端）
- **C. 作品集驱动**（约束：项目卡片 + 详情页）

拷问过程中 Q3 用户主动扩展 + Q5 推导后，形态升级为 **A + B + C 三层一体**（about + 项目 + 博客），实际更接近 Stage 1 D 混合综合站，但项目权重最高、博客偏项目复盘。

## 高危风险（必须在 SPEC 中处理）

- [x] **时间冲突**："1–2 晚 MVP" + "长期高频维护 + 博客"
  - **缓解**：采纳 (α) 方案 —— MVP 砍到「A 骨架 + 1 张真项目卡（vibe-coding-lab 占位）+ 1 张 'More coming' 占位 + 1 篇 hello/intro 博客 + 空博客索引」即可上线，后续按月迭代补内容。

- [x] **作品集 0 素材**：用户没有可立即展示的项目
  - **缓解**：MVP 用 vibe-coding-lab 自己作为第一张卡（meta，也是当前最有故事感的），加 1 张 "More coming" 占位。未来 3–6 月预期补 ~3 张。

- [x] **月更博客承诺与持续生产能力的差距**
  - **缓解**：节奏写为「理想 / 不是契约」。前期目标 1–2 篇/月，落空不算失败；frontmatter `pubDate` 透明化时效。

- [x] **网图头像版权风险**：用户 P2 答"网图"。直接搜索引擎扒图有侵权风险。
  - **缓解**：SPEC 阶段明确头像来源用免版权方案 —— 推荐 `dicebear` API（程序生成、零版权）/ Unsplash（CC0）/ boring avatars 库。**禁止**搜索引擎随手扒图。

## 中危风险

- [ ] Cloudflare Pages 国内访问偶尔慢 —— 可接受（已在 Q7 选 CF 时充分讨论）
- [ ] starter 模板魔改深度若失控会推翻 MVP 时间预算 —— SPEC Constraints 中限制"MVP 不动 starter 视觉，仅替换内容"
- [ ] 个人邮箱直挂会被爬虫扫 —— SPEC 要求 `mailto:` 链接 + JS 混淆 或 邮箱图片渲染
- [ ] 长期不更新会显尴尬 —— SPEC 要求项目卡 / 博客文章带 `pubDate`，超过 18 月项目自动 archived 灰显（二期）
