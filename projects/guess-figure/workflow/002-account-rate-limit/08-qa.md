## 质检就绪摘要（AI 填写）

### 改了什么

20 task 全完成（含 4 个 `fix(TX):` 回路）。核心改动：
- **D1 + 2 KV bindings** + migration 0001 创建 users/games 表（schema 预留 003 邮箱迁移字段）
- **cookie HMAC 鉴权层**（`src/lib/server/auth.ts` + `src/hooks.server.ts`）— 全局 `/api/*` 颁发签名 cookie + 滚动续期 365 天
- **限流 + LLM 成本兜底**（`src/lib/server/rate-limit.ts` + `src/lib/server/llm-cache.ts`）— 4 类 KV 计数器（IP/user/LLM-global/LLM-user）+ 缓存按 figure_id+aliases_hash 隔离
- **`/api/check-answer` 完整 pipeline**：normalize → server match-exact 短路 → KV cache → LLM；响应增 `cached / degraded / network_error` 三态
- **新端点** `/api/me`（D1 战绩汇总）+ `/api/game/finish`（幂等 INSERT）
- **前端识别 degraded/network_error**：`check-answer-client.ts` 在两种异常态**不消耗线索**；`AnswerInput.svelte` 加 200ms/5s 双阶段 loading；游戏结束自动 POST `/api/game/finish`

完整 commit 列表见 [07-implementation.md "进度"段](./07-implementation.md#进度与-06-tasksmd-同步)。

## Stage 8 人工 check 清单（用户填写）

### 1. AC1-3：cookie 颁发 + 篡改抵抗 + 滚动续期（GREEN 复核）

- [ ] **AC1 cookie 颁发**：Chrome 隐身窗口打开 [/play](https://guess-figure.pages.dev/play) → DevTools → Application → Cookies → 见 `gf_uid` 出现，且 HttpOnly ☑、Secure ☑、SameSite=Lax、Expires/Max-Age 约 365 天
- [ ] **AC2 cookie 篡改抵抗**：DevTools 改 `gf_uid` 值为 `00000000-0000-0000-0000-000000000000.tampered` → 刷新 → Console `fetch('/api/me').then(r=>r.json()).then(console.log)` → 看到的 user_id 与篡改前**不同**（被 server 拒后重新颁发新 user）
- [ ] **AC3 滚动续期**：上一步刷新后再点 Application → Cookies → 看 `gf_uid` 的 Expires 时间**比上次右移**（说明每次请求都续期 365 天）

### 2. AC8 / AC9：network_error / degraded 不消耗线索

- [ ] **AC8 network_error**（DevTools Offline 触发）：
  1. [/play](https://guess-figure.pages.dev/play) 开新一局，记录当前线索数（应该是 1）
  2. DevTools → Network 面板 → Throttle 下拉选 "Offline"
  3. 输入框输入"老丞相"（一个 alias 不在题库但语义合理的输入） → 提交
  4. 见错误提示"AI 响应异常，请稍后重试"（或类似 OQ3 文案）
  5. **检查**：左侧线索数应**仍是 1**（未消耗）；提交按钮可立即再点
  6. Network 面板恢复"Online"，再试一次正常提交，验证恢复

- [ ] **AC9 degraded 不消耗线索**（依赖 AC6/7 触发降级；建议合并测试）

### 3. AC6 / AC7：V/X budget 降级（需临时改 env vars）

> ⚠️ 此项需要临时改 wrangler.toml 的 env vars 触发降级，测完恢复。**或者跳过**——如果你接受单测覆盖（rate-limit.test.ts 已验证逻辑），可标 SKIP。

可选触发步骤（建议在 Stage 9 验收前一并做）：

- [ ] **AC6 单 user X budget 降级**：
  1. wrangler.toml 改 `LLM_BUDGET_PER_USER = "2"` → `git push`
  2. 等 CF Pages build 完成（~1-2 分钟）
  3. 浏览器同一窗口（同 cookie）连提交 3 次非 exact match 输入（如"老丞相" / "李大将军" / "孔大人"）
  4. **检查**：第 3 次响应：
     - 弹"今日 AI 裁判额度已用完..."提示
     - **线索数不变**（不消耗）
     - DevTools Network 看响应 body 含 `"degraded": true`
  5. wrangler.toml 改回 `LLM_BUDGET_PER_USER = "50"` → push 恢复

- [ ] **AC7 全站 V budget 降级**（同 AC6 套路，改 `LLM_BUDGET_DAILY = "2"`；测完同样恢复 `"8000"`）

### 4. AC11 / AC13 / AC20：cache hit 体感（性能 + 真命中）

- [ ] **AC11 + AC20 cache 体感**：
  1. 浏览器输入框输入一个**新的、稀有的、不在 aliases 中**的输入（如"亮先生"）
  2. 提交 → 第一次：DevTools Network 看 `/api/check-answer` 响应时间应 ~3000ms（LLM 调用）
  3. **等 70 秒**（让 KV 写完 + negative cache 过期）
  4. 同窗口同一输入再提交 → DevTools Network 看响应时间 < 1000ms 且响应体含 `"cached": true`
  5. 体感对比：第二次"瞬间"或近瞬间

- [ ] **AC13 cache hit p95 < 200ms（server-side）**：
  1. 接 AC11 测完后状态（cache 已写入），同输入连提交 5 次
  2. DevTools Network → 每条 `check-answer` 行点开看"Timing"标签
  3. **关注 server timing**（"Waiting (TTFB)"减去 RTT 估算）—— 看 server 处理时间是否 < 200ms
  4. 注意：浏览器看到的总时间 = server 处理 + 国际 RTT 500-700ms，纯 server 段 < 200ms 即 SPEC C9 PASS
