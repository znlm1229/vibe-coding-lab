# Stage 1 ｜ Brainstorm 头脑风暴

> 详细规范见 [workflow-spec](../../../../workflow-spec/specification.md#阶段-1--brainstorm-头脑风暴)
>
> **要点**：发散 3–5 个方向；每个配一句核心思路 + 一句主要权衡；**不评判出胜者**（由用户决定）。

## 任务

引入 guess-figure 第 4 种玩法 **海龟汤模式（Lateral Thinking Puzzle）**：用户提问 → AI 仅答「是 / 否 / 无关」→ 多轮后用户给出最终答案 → AI 判断对错。

> 用户原话：「开创新玩法：海龟汤模式。由用户输入问题，AI 回答，最后用户输入答案，判断是否正确。最关键的步骤在于 AI 回答用户输入的问题，只回答是与否，这里我们采用 rag 模式，将大量的史料数据塞入向量数据库，然后检索召回最后 AI 总结答案」

用户已锁定 **RAG 主路径**（向量库 + 召回 + LLM 三态总结）。本阶段在「玩法 × 技术深浅 × 整合方式」三维度交叉发散 5 个方向，**不评胜者**，留给 Stage 2 拷问选向。

## 方向

### A — 「人物海龟汤·最薄一层」复用 003 题库 + profiles 当 RAG 源  ❌ 已裁剪

- **核心思路**：人物池直接复用 003 的 65 人；每人 8-section markdown profile 当 RAG chunk（无需额外采集）；三态（是/否/无关），无轮次上限；完全独立 `/turtle-soup` 路由；向量库选 **SQLite + sqlite-vec**（依附现成 CF D1）或 **CF Vectorize**（生态贴合）；embedding 用 `qwen-embedding-v3`（中文专用 + 成本低）。
- **主要权衡**：上线最快（≈ 2 周），先验证玩法可玩性，复用已有数据资产成本极低；**但** profile 信息密度有限（每人 8 sections 偏简，~2-3K 字），冷僻提问（如「X 受过伤吗」「X 跟某历史事件有关吗」）易召不回 → AI 倾向答「否」造成误导。65 人池子玩家「老朋友再玩一遍」新鲜感打折。

### B — 「典故海龟汤·策展型」放弃猜人，挑 20-30 个戏剧性谜题型故事  ❌ 已裁剪

- **核心思路**：跳出「猜人」框架，做真正意义的海龟汤 —— 谜面如「项羽为何不渡乌江」「赵高指鹿为马时谁站出来反对」。每题人工策展 ~2K 字背景史料 + ~1K 字「核心 / 关键线索 / 常见误猜」标注；RAG 源就是这批策展文本（高密度高准）；三态 + 一段终极「故事还原」当底牌；独立 `/turtle-soup` 路由。
- **主要权衡**：玩法纯度最高（真正"海龟汤"质感），策展密度高 → 召回与三态判定准确率显著高于 A；**但**内容生产慢（每题人工 2-3K 字 ≈ 10x 003 单题），20-30 题策展投入 ~30-60 小时人工，可玩量有限，扩展靠人脑。

### C — 「混合双模式」嵌入主游戏，猜不出时切海龟汤辅助  ✅ 保留（= 子模式 1）

- **核心思路**：保留 guess-figure V3 主游戏（7 线索猜人），玩家用完线索且仍未猜出时，UI 出现「切海龟汤模式 · 还能问 5 个是/否」按钮；切过去后复用同一目标人物 + 复用 D 多源向量库做 RAG；同 1 道题双模式联动；后端 `games` 表加 `turtle_q_count` + `turtle_used` 字段，排行榜对「用过海龟汤」做得分折扣。
- **主要权衡**：粘性 + 教学价值最高（玩家学会"该问什么 yes/no 问题"），同时给死局兜底降低弃局率；**但** UX 复杂（两套提示器协调 + 折扣得分规则），耦合两套游戏后任何主游戏改动都要同步 retest 海龟汤侧，开发面 ≈ 1.5-2x A，回退困难。

### C2 — 「经典海龟汤·1 困难线索 + 限轮次」 ✅ 保留（= 子模式 2，用户 Stage 1 新增）

- **核心思路**（用户原话）「只给一条特别模糊困难的线索，然后用户来提问，并在最大指定数量内猜出」—— 独立 `/turtle-soup` 路由；每局开局只展示**一条策略性挑选的极困难线索**（"汤面"），玩家通过若干轮"是/否/无关"提问搜寻信息，必须在**指定最大轮次内**给出最终答案（"汤底"），超限或猜错算输；人物池复用 003 的 65 人 + D 多源向量库做 RAG。
- **主要权衡**：最接近经典海龟汤体验（汤面 + 提问 + 汤底三件套），玩法张力强、replay 价值高；**但**「困难线索」的"难度尺度"需明确定义（是从 003 现有 d6/d7 线索里挑？还是另写更隐晦的？）、最大轮次怎么定（10/20/无限？）、答错后能否继续问、计分规则、UI 与子模式 1 的差异化都是 Stage 2 待拷问的具体设计点。

### D — 「正史向量库·战略基建版」一次性建二十四史 + 维基大库  ✅ 保留（必建 — 用户拍板）

- **核心思路**：RAG 源**三层**：（1）65 profile markdown + （2）二十四史 Wikisource 全文人物相关篇章 + （3）维基百科主条目。向量库一次建好（~10-20K chunk）；embedding 用 `qwen-embedding-v3` 或 `bge-m3`；CF Vectorize（首版 < 100K 向量免费额度内）。这套 RAG 基础设施**不仅供 004**，还能反哺 006（自动给候选人物找史料 → 题库扩到 200）、未来 V5 做"人物百科内嵌问答"。
- **主要权衡**：战略价值最高（一次基建多任务复用），召回质量预期最高（多源 + 高密度）；**但**首版上线最慢（≈ 3-4 周，要先治理文本 / 切 chunk / 测准召回），首次嵌入一次性成本 ~¥10-20，CF Vectorize 月费 $5 / 百万向量（在 free 内）；用户已明确「**D 必须要建，多个数据源来充实向量库**」。

### E — 「无向量库·长上下文直答」跳过 RAG 工程，LLM 全文上下文  ❌ 已裁剪

- **核心思路**：把人物 profile + 维基摘要全文（共 ~3-5K tokens）每次都塞 LLM context，prompt 让模型「你只知道这些信息，回答是/否/无关」。每次提问 = 1 LLM call，零向量库、零 embedding、零基础设施；可上 `deepseek-v3.2`（128K ctx）或 `gemini-flash`（1M ctx）。
- **主要权衡**：工程量最小（≈ 1 周可上线），无新 dep 无新月费；**但**每次都打满 context，成本 ~¥0.004/题（不算贵，但用户量起来后不可控），且无法扩展到长文本（一旦想加二十四史立刻不够）；**核心矛盾：违背用户原话「采用 rag 模式」**。

## Stage 2 grill-me 拷问目标（待 Stage 2 用 grill-me skill 展开）

A/B/E 已删，原 A vs D 已不拷问。新拷问重心:

1. **D 基建·数据治理**：二十四史哪些篇章入库（仅 65 人涉及的传记？还是全 24 本？）、chunk 大小（段落 / 句子 / 滑动窗口）、embedding 模型 final 选（qwen-v3 vs bge-m3 vs gemini-embedding）、cache 策略（按 figure × question hash？）、首次嵌入成本 / 月费 / 性能 SLO
2. **子模式 1·游戏融合**：「切海龟汤」按钮何时出现（仅 7 线索用尽？还是任意时刻可切？）、得分折扣公式、`games` 表 schema 变更、跟 002 KV 限流 / LLM 预算的交互、UI 状态机
3. **子模式 2·新玩法定义**（核心拷问区）：「困难线索」难度尺度定义（用 003 现有 d7 线索？还是新生成？质量门槛？）、轮次上限 N（10/20/无限选哪个）、答错后能否继续 / 是否立即结束、计分（仅成败 or 含轮次效率）、UI 流程
4. **共享层边界**：两子模式共享 RAG 引擎到什么粒度？prompt template 共享还是各自定制？是/否/无关 三态判定逻辑统一还是分化？
5. **成本边界 & SLO**：单次 RAG（embedding + 向量召回 + LLM 三态总结）目标延迟 / 成本上限、KV cache 命中率目标、超出 002 日预算时降级策略
6. **Stage 3 prototype 范围**：spike 必须覆盖（a）D 召回准确率 — 拿 1 个测试人物，10 条提问测三态命中率；（b）prompt + 召回片段在两子模式下都 work；不需 spike 完整 UI
